# Hermes ↔ Time (Т‑Банк) — безопасная корпоративная интеграция

**Дата:** 2026-06-12
**Статус:** Реализовано (с поправкой по транспорту)
**Автор:** densin

> **ПОПРАВКА (по итогам живого запуска против `time.tbank.ru`):** Time
> оказался **Mattermost‑based**, а не Slack‑совместимым (REST = Mattermost
> v4 `/api/v4/`; сигнатура ошибки `api.context.404.app_error`). Поэтому
> `TimeAdapter` подклассирует `MattermostAdapter`, а не Slack. WebSocket у
> Time заблокирован прокси (HTTP 403), поэтому входящие идут через
> **REST long‑polling** — это и есть «long‑polling‑фолбэк», заложенный ниже.
> **Модель безопасности (corp_safe + hard‑deny + cron) не изменилась** —
> она транспортно‑независима и ключуется по идентичности `Platform("time")`.
> См. `plugins/platforms/time/README.md` и память `time-is-mattermost`.

## 1. Цель

Подключить агента Hermes к корпоративному мессенджеру **Time** (Т‑Банк) и
ограничить агента так, чтобы он **физически не мог выполнить опасное
действие** (выполнить произвольную команду в системе, удалить/перезаписать
файлы, управлять физическими устройствами и т.п.), **сохранив при этом
ядро** функциональности Hermes: чат, навыки (skills), память, поиск по
прошлым диалогам, веб‑поиск и анализ изображений.

Целевой сценарий первой итерации — **локальный прототип**, который затем
переносится в корпоративный контур.

## 2. Ключевые факты (по итогам разведки кода и API)

1. **Time API обратно совместим со Slack.** По публичным заявлениям Т‑Банка
   мессенджер Time переносит Slack‑ботов «без единой строки кода». Значит,
   адаптер Time строится **как тонкий подкласс существующего
   `gateway/platforms/slack.py`** с переопределением базового URL и
   токенов — а не пишется с нуля.
2. **Транспорт.** Slack‑адаптер уже поддерживает два режима:
   - **Events API + webhook** (предпочтительно) — входящие через публичный
     HTTP‑endpoint.
   - **Socket Mode (websocket)** — исходящее websocket‑подключение, не
     требующее публичного webhook‑URL. Это наш **фолбэк**, когда webhook
     запрещён политиками безопасности контура.
3. **Движок ограничения инструментов уже существует.**
   `model_tools.get_tool_definitions(enabled_toolsets, disabled_toolsets)`
   собирает схемы инструментов по toolset'ам. Ограниченная сессия:
   - не получает JSON‑схему отключённого инструмента (модель его «не видит»);
   - не может вызвать его и через мост `tool_search`/`tool_call` —
     `handle_function_call` содержит defense‑in‑depth, отклоняющий любой
     инструмент вне scope сессии.
   Это даёт **настоящую физическую гарантию**, а не просьбу к модели.
4. **Уже есть toolset `safe`** (`toolsets.py`): `web` + `vision` +
   `image_gen`, без терминала. Берём его за основу «корпоративного» профиля.

## 3. Классификация инструментов

**Опасные → запрещаем полностью** (источник физического вреда):

| Toolset | Инструменты | Чем опасен |
|---|---|---|
| `terminal` | `terminal`, `process` | произвольные shell‑команды |
| `file` | `write_file`, `patch` (запись) | перезапись/порча/удаление файлов |
| `code_execution` | `execute_code` | произвольный Python = обход всего |
| `computer_use` | `computer_use` | контроль рабочего стола |
| `browser` | `browser_*` | автоматизация браузера, действия от лица юзера |
| `homeassistant` | `ha_call_service` и др. | управление **физическими** устройствами |
| `delegation` | `delegate_task` | сабагент может вернуть себе опасные тулзы |

**Ядро → сохраняем:** `messaging` (чат через Time), `skills`,
`memory`, `session_search`, `web`/`search`, `vision`, `todo`, `clarify`,
`cronjob`. Чтение файлов (`read_file`, `search_files`) — **обсуждается**
(см. открытый вопрос O‑1): полезно для навыков, но даёт чтение ФС.

## 4. Архитектура

Два независимых, слабо связанных компонента. Оба — **плагины/конфиг, без
правок ядра Hermes** (по гайду `ADDING_A_PLATFORM.md`, plugin‑путь).

### 4.1. Адаптер платформы Time — `plugins/platforms/time/`

```
plugins/platforms/time/
  plugin.yaml      # метаданные, requires_env/optional_env
  adapter.py       # TimeAdapter(SlackAdapter): base_url, токены, enum Platform.TIME
  __init__.py
```

- `TimeAdapter` наследует Slack‑адаптер и переопределяет:
  - базовый URL Web API (`slack_bolt` `AsyncApp(base_url=...)`) →
    `TIME_API_BASE_URL`;
  - источники токенов: `TIME_BOT_TOKEN`, `TIME_APP_TOKEN` (Socket Mode),
    `TIME_SIGNING_SECRET` (webhook);
  - `Platform.TIME` (регистрируется через `register_platform()`, без правки
    ядрового enum — реестр поддерживает плагинные платформы).
- **Выбор транспорта:** если задан webhook (signing secret + публичный
  URL/intents) — Events API; иначе, если задан `TIME_APP_TOKEN` — Socket
  Mode. По умолчанию для корпоративного контура — Socket Mode.
- Авторизация пользователей, доставка cron, форматирование сообщений,
  typing‑индикаторы — наследуются от Slack‑поведения «бесплатно».

### 4.2. Профиль безопасности — toolset `corp_safe`

- Новый toolset `corp_safe` в `toolsets.py` (ядро без опасных тулзов).
- Применяется к Time‑сессиям **двумя слоями** (defense in depth):
  1. `enabled_toolsets = ["corp_safe"]` — модель видит только безопасные
     схемы;
  2. **жёсткий `disabled_toolsets`** (terminal/file/code_execution/
     computer_use/browser/homeassistant/delegation), применяемый
     безусловно, чтобы ни конфиг, ни плагин, ни сабагент не смогли вернуть
     опасные инструменты.
- Точка применения: конфиг сессии gateway для платформы Time
  (`gateway/session*.py` / `config.py`), плюс глобальный «hard deny» список
  как страховка. Конкретную точку фиксируем в плане реализации после
  трассировки `enabled_toolsets` от адаптера до `get_tool_definitions`.

## 5. Поток данных

```
Time (Slack‑совм. API)
  └─(Socket Mode wss / Events webhook)→ TimeAdapter
        → gateway session  (enabled=corp_safe, hard‑deny dangerous)
        → agent loop → get_tool_definitions(...)  ← опасные тулзы отсутствуют
        → ответ → TimeAdapter.send → Time
```

## 6. Обработка ошибок

- Нет токенов/URL Time → адаптер не стартует, понятная ошибка в
  `hermes gateway status` (через `requires_env`).
- Несовместимость конкретного Slack‑эндпоинта в Time → логируем и
  деградируем (фиксируем список фактически поддерживаемых методов в тестах).
- Если опасный инструмент как‑либо просочился в scope — defense‑in‑depth в
  `handle_function_call` его отклонит (это покрываем тестом).

## 7. Тестирование

1. **Юнит:** `corp_safe` не содержит ни одного опасного инструмента
   (assert против явного списка опасных tool‑имён).
2. **Защита:** сборка сессии Time → `get_tool_definitions` не отдаёт
   `terminal/process/execute_code/...`; прямой `handle_function_call`
   опасного тулза в Time‑scope → отказ.
3. **Адаптер:** мок Slack‑совместимого сервера (base_url подменён) —
   приём входящего и `send` исходящего; выбор транспорта webhook↔socket.
4. **Smoke:** локальный `hermes gateway` с `Platform.TIME`, обмен
   сообщениями против мок‑сервера.

## 8. Вне scope (YAGNI на первой итерации)

- Реальные креды боевого контура Time (нужны от заказчика).
- Голосовые сообщения/стикеры/вложения Time сверх Slack‑дефолтов.
- Sandbox/Docker‑изоляция и allowlist‑команды — не нужны, т.к. выбрана
  стратегия **полного запрета** опасных тулзов, а не их ограниченного
  разрешения.
